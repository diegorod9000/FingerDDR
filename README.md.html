**6.08 Team 43 Final Write-Up: FDDR** # Overview Our 6.08 final project is FDDR, or Finger Dance Dance Revolution, a competitive, music-based game brought to a pocket-sized platform. The essence of the game is as follows: players press up, down, left and right buttons on their ESP32 that correspond to dance moves indicated on their LCD screens. The dance moves are represented by arrows on the screen that move in time with the beat of the selected song. FDDR has preloaded songs from multiple genres for all players to enjoy. In addition, FDDR comes with an online leaderboard, where users can compare their daily scores with other players across the world for each song. All in all, Finger Dance Dance Revolution is a fun handheld video game that can bring people together and ease our restless fingers through an online platform. # Game Play ## Game Instructions The premise of FDDR is for a player to correctly time their button hits with the arrows appearing on the screen. To do so on our small device, players will utilize the four buttons connected on the breadboard, the left, down, up, and right buttons. Once the player selects a song from the menu, they are taken to a waiting screen on the device while the player navigates to the play button on our server. After pressing play, the screen will begin to display our game arrows, and the player then is challenged to time their button presses with the arrow reaching the bottom red line. The closer the arrow is to our red line, the more points the player receives. If the player misses the arrow, points are deducted from the score. ## FDDR Tutorial The player is greeted with our main menu screen, at which point they can either click the left button to check the current FDDR leaderboard, or click the right button to play a game. When selecting songs from our game menu, use the joystick to navigate up and down through the list, before pressing the right button. Then, the player is taken to a name input screen, where they can choose their username for this round. After doing so, the game is set to be in waiting mode, until three beeps are heard from the server side. This indicates to the ESP32 that the game is now ready to be played and the music from the server will be in sync with the arrows on our device. Once the game is completed, the player is brought to the songâ€™s current leaderboard to compare their results with other competitors. The player then has the option to play another song with the same username, or return to our main menu. # Functional Block Diagram of the System Below is the state machine diagram for our system: ********************************************************************************* * .-------------------------------------. * * | | * * v | * * left button .--+---. right button | * * .-------------+ +----------------. | * * | | Menu | | | * * | .--------->+ +<-----------. | | * * | | '------' | | | * * | |left button left button| | | * * | | | | | * * v | | v | * * .-+--+-. .-+---+-. | * * | Leader +--------------------------->+ Choose | | * * | Board | right button .-->+ Song | | * * | | && not from menu | '+----+-' | * * '-+----' | | | | * * ^ '----' |right button | * * | joystick up/down v | * * | .---+---. | * * | | Enter | | * * | .-->| Name | | * * | | '+----+-' | * * | joystick up/down| | | | * * | '----' |right button | * * | v | * * right| .--------. .-------. .--+---. | * * button| | Finished | | Playing | | Play | | * * '-+ Song +<-----+ Song +<-------+ Song +--------' * * '--------' song '-------' beeps '------' beeps not heard * * ends heard within 10 sec * * * * * ********************************************************************************* # State Machine Our state machine setup is designed to make the playthrough experience as simple as possible while navigating each component of it, and offering some flexibility for mishaps or missed inputs. In CHOOSE_SONG, the song names are stored in a list that rotates as the joystick is moved up or down. It shows the current song in a larger text, and then the previous song above and the next song below in smaller text (wrapping around, not ending at the beginning/end). For the name, it highlights the current letter the user is on, and they can scroll up or down through the alphabet with the joystick. When they want to move on to the next letter, they can move the joystick to the right and it will highlight the next letter, starting at A again, showing that this is the letter being chosen now. When the user finishes a round and chooses another song straight from the leaderboard, the username is saved from before and they donâ€™t have to submit their name again. ````` case ENTER_NAME: set_screen(ENTER_NAME); // right button pushed -> play song if (right_reading != old_right_reading) { if (!right_reading) { Serial.println("right button pressed, going to play song"); strcat(username,alphabet[alphabet_index]); // append the last letter that you chose tft.fillScreen(TFT_BLACK); tft.setTextColor(TFT_YELLOW); tft.drawCentreString("READY FOR BEEPS",SCREEN_WIDTH/2, SCREEN_HEIGHT*2/5,2); page_state = PLAY_SONG; waiting_timer = millis(); delay(200); } } // joystick to the right -> choose next letter else if (x_reading > JOYSTICK_MAX*0.75) { Serial.println("joystick right, moving to next letter"); name_index++; strcpy(chosen_letter,alphabet[alphabet_index]); alphabet_index = 0; delay(400); } // joystick down -> next letter in alphabet else if (y_reading < JOYSTICK_MAX*0.25) { Serial.println("joystick down, choosing next letter"); alphabet_index = mod(alphabet_index+1, 26); delay(300); } // joystick up -> previous letter in alphabet else if (y_reading > JOYSTICK_MAX*0.75) { Serial.println("joystick up, choosing previous letter"); alphabet_index = mod(alphabet_index-1, 26); delay(300); }break; ````` When a song is selected, the function getNotes() is run, which takes the songâ€™s name as a parameter. The script runs all necessary GET requests and loads the song arrow data into the vectors to be used once the song starts. ## Map Arrows In order to draw small arrows floating down the screen as in typical Dance Dance Revolution, we used triangles! By drawing small triangles, and redrawing them lower every 25 milliseconds and covering the last one, there is an illusion that arrows are coming down the screen. At first, we used global variables to draw these triangles and ran into bugs with having multiple triangles on the screen at one time. In order to solve this, we used multiple vectors to hold the values of the current screens drawn arrows. There is a left, down, up, and right vector for each section of the screen. The values in these vectors are incremented similarly to before, and the arrows on the screen are drawn using these as corner values. If one of the values in the triangle in a vector exceeds the screen limit, it and its triangle corner partners are removed from the vector all at once to avoid messing up the amount of numbers in the vector. ## Scoring and Hit Detection The same button detection used in the state machine is used to detect if one of the 4 directional buttons has been pressed. The score variable was added, and is displayed on the screen. There are 3 scenarios that affect the score variable: pressing a button when there isn't a corresponding arrow on screen, pressing when there is, and failing to press the button in time. * If a directional button is pressed and there isn't a corresponding arrow on screen, 50 points are deducted. * If an arrow reaches the end of the screen without a button press, 50 points are also deducted after it is removed. * If there is an arrow when the button is pressed, the score added is directly proportional to how close the arrow is to the end of the screen. After the corresponding button is pressed or the arrow reaches the end, the arrow is removed. If multiple of the same arrow are on screen during a button press, only the arrow closest to the bottom is removed. Only the button being pressed down will constitute a button press, not the button being held nor the button being released. This scheme of scoring encourages the player to only press if there is an arrow on screen, try to maximize their score by hitting as accurately as possible, but also being cautious about missing notes. Upon setting up the map, the score is reset to 0. # Integration between Arduino and Server Side ## Database The database is held on the server, called `lboard_final.db`. This database stores 4 attributes for every entry: the username, the song that was played, the score they earned, and the time at which they finished the game. This is updated via POST request each time a player finishes a game. This allows for the user to see all top scores within 24 hours via GET request to `lboard.py` (which happens from the home screen on Arduino) or top scores by specific song (which displays after a player finishes a game, as well as at the website shown above in the leaderboard section). Here is what `lboard_final.db` looks like: | username | song | score | time | |-------------|----------|----------|------------------| |FDDRISLIFE |Rasputin |1860 |2021-15-5 10:43:05| |-------------|----------|----------|------------------| |ILUVFDDR |Castaways |2100 |2021-16-5 17:24:55| ## Music Syncing In order for the music on the server to sync with the game on the Arduino, we decided to use the microphone to detect a starting signal attached to the start of each song. The starting signal is three beeps at 500Hz range. In order to detect only these beeps in a system with background noise, we tried both a band pass filter and FFT to isolate sounds between 490-510Hz. With this library, we got the frequency of the sounds coming into the microphone, and only made the machine respond if the peak frequency heard was between the desired range. The machine doesnâ€™t respond to background noise, rubbing the microphone, talking, or any other frequencies in the 1-20000 Hz range. # Server Side ## Homepage The homepage is located at http://608dev-2.net/sandbox/sc/team43/fddr/game.py. It contains a button directed to the leaderboard page and an audio player. The audio player is synced to the song that the user has selected on the Arduino. When the user enters the `Play Song` state (see state machine diagram), the Arduino posts their selected song to the server code. Then, the server code stores the song in the `current_song` database. When the homepage is refreshed, the server reads the most recent song from the database and plays its audio file modified with the three starting beeps. If the song is invalid, the audio player is grayed-out and cannot be played. ## Leaderboard The leaderboard page is located at http://608dev-2.net/sandbox/sc/team43/fddr/leaderboard.py. It uses the database `lboard_final` populated by the Arduino as described in the Database section below. In order to display the top scorers for each song, the server code first queries for distinct songs within the database. Then, it stores in the `sql_query` dictionary the top 5 (or less, if fewer than 5 users played that song) `(user, score)` pairs for each song. Each distinct song is made into its own HTML table and displayed on the leaderboard page. The page also has a button to return to the homepage. ## Music Generation To generate music, the file is first downloaded into our library as a wav file. The file audio_editor.py to alter these wav files. First, the wav file is translated into a dictionary representation. To add the three beeps to each song, a beep sound wav file was downloaded, translated, and then repeated three times with about .25 seconds in between each sound. This new sound dictionary then gets added to the sound dictionary of the song wav file. This creates the new sound that will play on the server, which is cut off after a minute to keep gameplay in sync. ## Map Generation First, a map is generated as a txt file, which communicates the bpm of the song, as well as the relative location of each note in the song using beats. This can either be done manually, or more easily, using a script that writes the txt file. The get_notes.py file will load the map into the database prior to it being loaded into the actual game. Using the file map_creator.py, the script will convert the txt file data into a list which contains each note, and the time in milliseconds before it is supposed to appear on screen. This is then put on a separate database using SQL, where it can be acquired using a GET request (because the data is very large, in order to save space, 4 requests are required. Each one will return all notes on a single lane.) When the main script enters the play state, it will run a function that does all 4 GET requests, attributed to the specific song that will be played. This makes sure that each song has a map synchronised to its bpm, and will appear as intended. # Documentation ## Parts List We used one part for this project, this joystick & button combo (B07QQXWBCF): [https://www.amazon.com/Magic-shell-Joystick-Expansion-Keyboard/dp/B07QQXWBCF] We used the joystick for the scroll-through functionality on the song-choice state and the name-choice state, and we used the buttons for navigating through the pages throughout the game, as well as playing the game itself. # Videos ## Week 1 We first created a song and notes database.
